Accessibility

Accessibility information for the current release.

The goal of this page is to provide concrete, public information about the app’s accessibility posture without overstating feature support that should be validated on real devices.

Email accessibility feedback Open support

Best practice

Accessibility claims should be based on common-task testing across every supported device class and the accessibility features people rely on.

Current posture

What this public page can say accurately today

Labeled controls

The app is designed around labeled controls and structured grouping across calculator, help, privacy, support, and history surfaces.

Scalable typography

The interface is designed to support larger text across top bars, input/output areas, settings, and overlays, though release claims should still be validated on device.

Appearance support

The current release supports light and dark appearance modes and aims for strong contrast and readability.

Touch clarity

Primary interactions are built around generous touch targets and visually grouped actions to reduce ambiguity.

How to evaluate

Common tasks matter more than isolated screens

1. Test the common tasks

Verify the things people actually download the app for: entering values, switching calculators, reviewing history, opening support/privacy pages, and completing the core calculations they depend on.

2. Test with the accessibility feature in use

Evaluate VoiceOver, Larger Text, Dark Interface, contrast, and other relevant settings while performing those common tasks.

3. Test on supported devices

Run those checks on the iPhone and iPad configurations the app supports, not just one simulator.

4. Keep the website accurate

If accessibility support changes, update both the app’s App Store labels and this public page so the public record stays truthful.

Reporting checklist

What helps accessibility reports get resolved

  1. Describe the device model and iOS/iPadOS version.
  2. Identify the calculator type or screen involved.
  3. State which accessibility feature was in use, such as VoiceOver or larger text.
  4. Describe the exact problem and the expected behavior.
